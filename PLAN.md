# What's Cookin' — Project Plan

An alternative, modern frontend for [Mealie](https://mealie.io) — the self-hosted recipe manager.

---

## Vision

Mealie is a great tool, but its built-in frontend can feel slow and dated. What's Cookin' aims to
deliver a fast, opinionated, client-side UI for Mealie instances, with a focus on:

- **Speed** — near-instant navigation via TanStack Router's preloading
- **Clean UX** — a recipe-first interface that gets out of your way in the kitchen
- **Deployability** — a pure SPA that can be hosted on any CDN with no server required
- **Type safety** — a fully generated, type-safe API client that tracks Mealie's OpenAPI spec

The app connects to any existing Mealie instance via a configurable `VITE_MEALIE_BASE_URL`. It does
not replace or modify the Mealie backend.

---

## Architecture

### Stack

| Layer | Choice | Rationale |
|---|---|---|
| Framework | [TanStack Start](https://tanstack.com/start) (SPA mode) | File-based routing, type-safe links, no SSR overhead |
| Language | TypeScript | End-to-end type safety |
| API client | [@hey-api/openapi-ts](https://heyapi.dev) + Fetch | Generated from Mealie's OpenAPI spec, zero runtime overhead |
| HTTP | Native `fetch` | No extra dependency; built into every modern browser |
| Package manager | pnpm | Fast, disk-efficient |

### Rendering mode

SPA mode is enabled (`spa: { enabled: true }` in `vite.config.ts`). On `pnpm build`, TanStack Start
prerendering generates a single `/_shell.html` file. All routing happens client-side. A CDN
catch-all redirect to `/_shell.html` is all that's needed for deployment.

### Authentication

Mealie uses OAuth2 password-flow (`POST /api/auth/token` with `application/x-www-form-urlencoded`).
The token response returns a bearer token that must be attached to subsequent requests.

**Current approach: in-memory only**

- The token is stored in a module-level variable in `src/api/auth.ts`
- A request interceptor in `src/api/client.ts` attaches `Authorization: Bearer <token>` to every
  outgoing request
- Tokens are **not persisted** to `localStorage` — this avoids XSS token theft at the cost of
  losing the session on page refresh
- A future iteration will add optional persistence (e.g. `sessionStorage` behind a "remember me"
  checkbox)

### API client

The client lives in `src/api/` and is structured as follows:

```
src/api/
├── generated/          # Auto-generated by `pnpm generate` — do not edit
│   ├── client.gen.ts   # hey-api fetch client instance
│   ├── sdk.gen.ts      # One typed function per Mealie API operation
│   ├── types.gen.ts    # All request/response TypeScript types
│   └── index.ts        # Re-exports everything
├── client.ts           # Configures baseUrl + auth interceptor on the generated client
└── auth.ts             # In-memory token store; login / logout / refresh / getCurrentUser
```

To regenerate the client after a Mealie API update:

```sh
pnpm generate
```

## Development

### Prerequisites

- Node.js 22+
- pnpm 10+
- A running Mealie instance (or use `https://demo.mealie.io`)

### Getting started

```sh
# Install dependencies
pnpm install

# Copy and configure environment
cp .env.example .env
# Edit VITE_MEALIE_BASE_URL in .env to point at your Mealie instance

# Start the dev server
pnpm dev

# Regenerate the API client (after Mealie API changes)
pnpm generate
```

### Environment variables

| Variable | Default | Description |
|---|---|---|
| `VITE_MEALIE_BASE_URL` | `http://localhost:9000` | Base URL of the Mealie instance |

> Variables are baked in at build time by Vite. For production, set them in your deployment
> platform's environment variable config, not in a `.env` file.

## Decisions log

| Decision | Rationale |
|---|---|
| SPA mode over SSR | No server needed; CDN-deployable; Mealie API handles all data |
| hey-api codegen | Types always in sync with the Mealie spec; eliminates hand-written boilerplate |
| Native fetch (no axios/ky) | Zero extra dependency; sufficient for our use case |
| In-memory token storage | Protects against XSS; session-scoped is acceptable for a kitchen app |
| Generated code committed | Makes the repo self-contained; avoids network calls in CI builds |
